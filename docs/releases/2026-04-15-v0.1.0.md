## 今天做了什么

### 1) Q1-Q7 架构硬化

今天最重要的一步，是把架构审计里的 Q1-Q7 真正落地：

- 新增 `core/validation.py`，把最容易污染 benchmark 的字段前移到 L0 物理边界校验。
- 新增 `ExtractionRun` append-only 审计表，为后面的 provenance 链路打地基。
- 新增 `scripts/export_verified.py` / `scripts/import_verified.py` / `scripts/sync_to_vps.sh`，明确 raw PDF 本地留存、verified 结构化数据可同步的三层数据模型。
- 新增 `scripts/nightly_burn.sh`，把夜间验证和维护脚本标准化，但避免自动提交破坏历史。
- 上传链路新增 PDF 魔数和尺寸校验。
- `report_parser/batch_jobs.py` 增加 TTL 清理逻辑，降低内存长期积累风险。
- `core/database.py` 与 `docs/runbooks/postgres_migration.md` / `docs/runbooks/vps_hardening.md` / `docs/policies/data_handling.md` 共同把“未来切 Postgres”的准备工作做好。

这意味着今天项目工程能力真正提升了，不再只是继续堆功能。

### 2) ExtractionRun 审计链路补齐

今天把“可追溯”从概念变成了实物：

- `scripts/seed_german_demo.py` 成功提取后会写入 `run_kind=extract` 的审计记录。
- `scripts/audit_extractions.py` 成功审计后会写入 `run_kind=audit` 的审计记录，附带 `prompt_hash`、`raw_response`、`verdict` 和摘要 notes。
- 后端恢复 `GET /report/{id}/audit-trail`。
- 前端 Company Profile 的 `PeerComparisonCard` 新增 Source trail 折叠区，用户可以直接看到 run kind、model、verdict、notes、created_at。

今天之后，我们已经能回答这些关键问题：

- 这条数据是谁写进去的？
- 是 extract run 还是 audit run？
- 用了哪个模型？
- 审计 verdict 是什么？
- 前端能不能直接看到？

答案现在都是：**能**。

### 3) 修复 signed EU Taxonomy 百分比语义

今天还处理了一个非常关键的真实数据问题。

在硬化后第一次跑 export 时，thyssenkrupp 2024 的 `taxonomy_aligned_capex_pct` 暴露出真实披露语义与旧校验逻辑不一致：披露值是 `-9%`，而旧逻辑默认所有百分比都必须在 `0..100`。

今天的修正包括：

- `core/validation.py` 将 taxonomy revenue/capex 两个字段改成支持 `[-100, 100]`。
- `tests/test_validation.py` 新增 signed 百分比正负回归。
- `scripts/fix_thyssenkrupp_capex.py` 改为恢复真实披露值 `-9.0`，而不是继续写成 `+9.0`。

这一步非常重要，因为它把项目从“为了让导出通过而临时归一化”推进到“保留真实披露语义，同时让验证与导出继续成立”。

### 4) 前端体验增强

今天前端的推进不是换皮，而是增强分析动作：

- Dashboard 支持 coverage 字段排序。
- 横向指标卡更适合概览。
- 新增 `CoverageFieldPage`，每个 coverage 字段都可以 drill-down 到公司排名表。
- Source trail 让公司页里的数据证据链可视化。

这些功能让项目更像一个“可解释的分析工具”，而不只是一个展示页面。
