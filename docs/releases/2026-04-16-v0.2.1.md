# ESG Research Toolkit v0.2.1 Release

**Release Date**: 2026-04-16
**Previous Version**: v0.2.0 (same day)
**Status**: ✅ Production Ready

## Overview

Follow-up release on the same day as v0.2.0. v0.2.0 delivered the multi-year
trend story; v0.2.1 hardens it with regression tests, operational scripts,
and a self-healing / self-polishing automation toolkit so future iteration
loops can run with minimal human babysitting.

The core goal: **make the multi-year trend capability durable and make the
project easy to run / verify / polish end-to-end with a single command**.

## Key Changes

### 🧪 Multi-year trend regression coverage

New pytest case `test_company_history_three_year_trend_ordering_and_yoy`
seeds three years (2022/2023/2024) for a synthetic company and asserts:

- `/report/companies/{name}/history` trend array is ordered ascending
- `trend` points carry normalised keys (`renewable_pct`, `scope1`, …)
- `CompanyProfile` exposes `years_available` = `[2022, 2023, 2024]`
- `latest_year = 2024`, trend tail is `2024` then `2023`

This is the regression net that guards the BASF / RWE / Deutsche Telekom
demo story against future refactors to the history aggregator.

**Result**: `OPENAI_API_KEY=dummy .venv/bin/pytest -q` → **120 passed**
(was 110 at the start of the day, 111 at v0.2.0 mid-point).

### 🛠 Automation toolkit (`scripts/automation/`)

Five new scripts + a single README index. Each one is idempotent, writes
structured logs, and is designed to be re-runnable without cleanup.

| Script | Purpose | Key output |
|---|---|---|
| `run_fullstack.sh` | Start / stop / status uvicorn + vite with health checks; supports `--detach`, `--stop`, `--status` | `logs/backend.log`, `logs/frontend.log`, `logs/fullstack.pids` |
| `auto_fix_smoke.sh` | Self-healing test loop: pytest + lint + build + smoke, on failure emits a Claude/Codex-ready fix prompt | `logs/autofix_prompt_<ts>.md` |
| `interactive_dev.py` | Menu-driven helper: DB summary, API health sweep, multi-year trend peek, pytest, seed status, recompute, screenshots | `logs/interactive_log.md` |
| `ui_autopolish.py` | Playwright desktop + mobile screenshots → vision-LLM aesthetic critique → auto-appended task list | `screenshots/<ts>/`, `ui_reports/<ts>/critique.md`, `docs/exec-plans/ui_autopolish_tasks.md` |
| `stress_test.sh` | API load test (hey / ab / curl-loop fallback), frontend page sweep, rate-limit probe on `/report/upload` | `logs/stress_<ts>.md` |

Design principles (see `scripts/automation/README.md`):

1. **Idempotent** — repeated runs never corrupt local state; `run_fullstack`
   reuses an already-listening port
2. **Traceable** — every script writes timestamped markdown / JSON under
   `scripts/automation/logs/`
3. **Composable** — script output is structured so it can be pasted straight
   into an AI session as context
4. **Self-healing friendly** — on failure, output is a ready-to-paste fix
   prompt rather than a raw stack trace

### 🌱 Seed pipeline ergonomics

`scripts/seed_german_demo.py` gains three new filter flags so you can
iterate on a single company without reprocessing the full 20-company
manifest:

- `--only rwe-2024` / `--only "BASF SE"` (accepts slug or company name,
  repeatable, comma-separated values supported)
- `--slug rwe-2024 --slug basf-2024` (legacy explicit slug filter)
- `--company "RWE AG" --company "BASF SE"` (legacy explicit name filter)

Empty selections fail fast with a clear message; `--only` cannot be
combined with the legacy flags.

Coverage backed by 4 new unit tests in `tests/test_seed_german_demo.py`
(filtering behaviour, empty-selection rejection, mutually-exclusive flag
validation).

Also in this area:
- `phase_b` now drops `"no concern"` placeholder concerns before writing
  the anomalies report, so validation runs stay clean
- Runbook updated with a low-cost small-sample verification snippet using
  `--only`
- `OPENAI_VALIDATION_MODEL` default bumped to `gpt-5.4-mini` (runbook
  wording aligned)

### 🔍 Upload evidence fallback

`report_parser/api.py` introduces `_upload_evidence_summary()` which
prefers evidence already produced by the analyzer and only falls back to
a per-metric placeholder (with `source_type=pdf`, `file_hash=…`) for
metrics that are non-null in the extracted `CompanyESGData`. Previously
the upload endpoint hard-coded three metrics regardless of what the PDF
actually contained.

Two new unit tests pin the "prefers analyzer evidence" and "fallback
skips null metrics" paths.

### 🧹 Repo hygiene

`.gitignore` additions:

- `scripts/automation/screenshots/` — Playwright captures
- `scripts/automation/ui_reports/` — vision critique markdown
- `scripts/automation/*.png` — ad-hoc captures

rule.

## Known Limitations / Follow-ups

- **Company-name canonicalisation gap** (carried from v0.2.0): extracted
  `SAP` vs manifest `SAP SE`, `Volkswagen Group` vs `Volkswagen AG` split
  multi-year records into two identities. Fix lives in
  `company_identity.py`; deferred to next iteration.
- **`ui_autopolish.py` LLM cost** is metered per run — use
  `--screenshot-only` when you only need fresh captures without critique.
- **2022/2023 coverage** still only covers 5 core DAX companies; BMW /
  Daimler / Henkel / Siemens / DHL historical manifest entries remain to
  be resolved (several URLs are bot-protected).
