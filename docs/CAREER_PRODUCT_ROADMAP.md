# ESG Research Toolkit — Career-to-Product Roadmap

> Last updated: 2026-04-13  
> Primary goal: turn this project into a high-credibility analysis-engine portfolio piece first, then expand it into a lightweight sellable product.

## 1. Why This Project Exists

This project is not just a generic ESG dashboard.

Its highest-value role is to connect:

- AI-assisted product building
- sustainability management knowledge
- public-disclosure analytics
- cross-framework regulatory reasoning
- a concrete portfolio story for internships and English-speaking roles in Germany

The project should help answer one clear question:

**Can we use public disclosures to explain how a company is progressing over time under changing sustainability rules, with evidence and structured comparison?**

If the answer is yes, the project becomes useful in three ways:

1. a portfolio project for internships and jobs
2. a differentiated analysis product
3. a foundation for future consulting or lightweight SaaS monetization

## 2. Product Positioning

### 2.1 Recommended Positioning

**AI-driven ESG regulatory intelligence platform**

Built from public disclosures, the platform should focus on:

- company history and trend analysis
- framework-version comparison
- evidence traceability
- cross-region regulatory mapping
- explainable scoring and change analysis

### 2.2 What This Project Should Not Try to Be First

Do **not** start by competing head-on with enterprise ESG workflow suites.

Avoid leading with:

- enterprise internal reporting workflows
- complex org hierarchies
- billing and payment systems
- approval chains and admin-heavy operations
- broad “all-in-one ESG ERP” ambitions

Those areas are expensive, crowded, and not the strongest job-signal for the current stage.

### 2.3 Recommended Wedge

The strongest wedge is:

**public-data-driven, evidence-backed, multi-period, multi-framework company analysis**

This is a better fit for:

- your sustainability background
- AI vibe coding execution
- English-speaking roles in Germany
- future consulting or API-based commercialization

## 4. The Core Strategy

### One-Sentence Route

First build a **high-credibility analysis engine portfolio piece**, then expand it into a **lightweight sellable product**.

First break through **data, versioning, evidence, and trends**, then add **login, permissions, payments, and client management**.

### Priority Order

1. history trends + evidence traceability + rule versioning
2. company profile page + showcase cases + English explanation
3. cross-framework mapping + change explanation
4. industry benchmark / peer comparison
5. API, export, and consultant delivery support
6. login, payment, permissions, and admin operations

## 5. Defensible Moat

The moat should not come from UI polish or standard SaaS plumbing.

The moat should come from the combination of:

- structured public-disclosure data assets
- reusable evidence graph
- framework-version-aware interpretation
- company change analysis over time
- explainable cross-framework mapping
- domain-specific judgment for EU and cross-border use cases

### Moat Principles

1. Every key number should be explainable.
2. Every analysis result should be reusable.
3. Every framework result should be tied to a version.
4. Every trend should be anchored in a reporting period.
5. Every showcase should tell a business and regulatory story, not just show charts.

## 6. What To Build First

### 6.1 Immediate Product Thesis

Build for one narrow but compelling promise:

**“Show how a company’s sustainability disclosure and regulatory readiness changed over time, and explain why.”**

### 6.2 MVP Scope

The first strong MVP should contain:

- 3 to 5 real companies
- 3 to 5 annual reporting periods each
- 2 high-value frameworks to start
- evidence drill-down
- trend visualizations
- one strong company profile page
- one comparison view
- one English product narrative page in the repository

### 6.3 Why Annual First

Public ESG data is most reliable at annual-report granularity.

Quarterly or monthly analysis can be added later, but the first trustworthy version should anchor on:

- annual sustainability reports
- annual reports
- major regulatory disclosures
- material event disclosures where relevant

## 7. 12-Week Roadmap

## Phase 1 — Foundation (Weeks 1-4)

### Goal

Create the analysis-engine core that is credible enough to show in interviews.

### Outcomes

- evidence-aware data model
- trend-ready storage model
- framework-version scaffolding
- first company profile experience

### Week 1: Data Model Upgrade

- Introduce explicit entities for company, reporting period, evidence, metric, derived result, and framework version.
- Separate raw extracted facts from interpreted analysis results.
- Define a stable internal schema for “source -> metric -> rule -> score -> explanation”.

### Week 2: Evidence Traceability

- Store source metadata for each extracted metric.
- Add source references such as document name, section, page, snippet, and extraction timestamp.
- Enable drill-down from displayed metric to evidence.

### Week 3: Historical Trend Layer

- Create APIs for per-company multi-year history.
- Support consistent trend computation for key metrics and taxonomy proxies.
- Add “change vs prior year” summaries.

### Week 4: Framework Versioning

- Add version metadata to framework definitions and derived outputs.
- Start with the currently supported core frameworks and make versioning explicit in APIs.
- Define how old analysis results remain readable after rule updates.

## Phase 2 — Showcase Layer (Weeks 5-8)

### Goal

Turn the engine into a portfolio-quality demo.

### Outcomes

- strong company profile page
- 2 to 3 polished public case studies
- English-first explanation of value and method

### Week 5: Company Profile Page

- Ship a detailed company page with trend charts, score summaries, evidence entry points, and progress interpretation.
- Make this page the portfolio centerpiece.

### Week 6: Comparison and Explanation

- Add comparison views for company vs company or company vs prior years.
- Explain what improved, what worsened, and what remains missing.

### Week 7: Public Case Studies

- Prepare a small set of real-company examples with clear notes on data origin, limitations, and interpretation logic.
- Write concise English narratives for each case.

### Week 8: Portfolio Packaging

- Refine README and screenshots.
- Prepare a short demo script and a 3-minute presentation flow.
- Add a “Why this matters” section aimed at recruiters and hiring managers.

## Phase 3 — Differentiation (Weeks 9-12)

### Goal

Move from a good portfolio to a differentiated niche product.

### Outcomes

- framework mapping and delta explanation
- peer benchmark starter set
- API/export hooks for future consulting usage

### Week 9: Cross-Framework Mapping

- Map comparable dimensions across frameworks.
- Show which parts are comparable, partially comparable, or not comparable.

### Week 10: Difference Explanation

- Add textual explanations for score deltas across frameworks and time periods.
- Highlight disclosure gaps versus rule changes.

### Week 11: Benchmark Starter

- Add a limited peer-comparison layer for a narrow sector slice.
- Focus on interpretability rather than broad market coverage.

### Week 12: Delivery Surface

- Add export/API support needed for consulting-style delivery.
- Keep account systems out unless there is real user pull.

## 8. Engineering Sequence

The recommended engineering order is:

1. data model for periods, evidence, metrics, derived analyses, and framework versions
2. evidence capture and source drill-down
3. company historical trend APIs
4. company profile page
5. framework version switch and result persistence
6. cross-framework mapping and explanation
7. peer benchmark starter
8. export/API delivery support
9. login and permissions
10. payments and customer operations

## 9. Success Criteria by Stage

### Stage A — “Strong Portfolio”

You can already use the project in applications when all of the following are true:

- deployed demo exists
- one company profile page feels complete
- at least 2 real company examples work end to end
- evidence traceability is visible
- trend view works across multiple years
- English README and project story are clear

### Stage B — “Credible Niche Product”

You can start testing monetization when:

- results are reusable across sessions and users
- framework-version comparisons are stable
- API or export workflows help consultants or analysts
- a narrow benchmark workflow exists for a target audience

### Stage C — “Product Expansion”

Only after earlier stages are stable should you invest more heavily in:

- user accounts
- team permissions
- subscription logic
- invoicing and billing
- admin operations

## 11. Risks and Mitigations

### Risk 1: Building Too Broadly

Mitigation:

- keep the first promise narrow
- resist building full enterprise workflows early

### Risk 2: Weak Data Credibility

Mitigation:

- show sources
- track evidence
- state limitations clearly

### Risk 3: Too Much Product Plumbing Too Early

Mitigation:

- delay auth, payments, and admin systems
- prioritize analysis differentiation

### Risk 4: Too Little Showcase Value

Mitigation:

- make one excellent company page
- prepare recruiter-friendly English explanation

The north star for the next milestone is:
